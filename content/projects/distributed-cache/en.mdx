## Problem

Our platform was handling 10M+ daily API requests, each requiring multiple database queries. The average response time was 320ms, and database load was approaching critical limits. We needed a caching strategy that could dramatically reduce latency without introducing consistency headaches.

The existing approach used a simple in-memory cache with TTL, but it was per-instance and inconsistent. Cache misses were expensive, and stale data was silently served.

## Approach

### Requirements

- **P99 latency under 50ms** for cache hits
- **Strong consistency** for critical data paths
- **Automatic failover** with no manual intervention
- **Cost-effective** — no new infrastructure budget

### Architecture

I designed a two-tier cache layer:

1. **Local tier (in-memory):** Per-instance LRU cache for hot keys. Sub-millisecond access.
2. **Global tier (Redis Cluster):** Distributed Redis cluster with read replicas for cross-instance consistency.

Keys were hashed and distributed across Redis shards. A background invalidation pipeline using Redis Pub/Sub ensured that writes to one instance invalidated stale entries everywhere.

### Consistency Model

We used a **read-through, write-invalidate** pattern:

- On read miss: fetch from DB, populate local + global cache
- On write: update DB, publish invalidation event, clear local cache
- TTL-based fallback: all entries expire after 5 minutes as a safety net

## Results

| Metric | Before | After | Improvement |
|--------|--------|-------|-------------|
| P99 latency | 320ms | 28ms | **91% reduction** |
| P50 latency | 145ms | 4ms | **97% reduction** |
| Cache hit ratio | — | 94% | — |
| DB CPU load | 78% | 22% | **72% reduction** |
| Monthly infra cost | $12,400 | $8,100 | **35% savings** |

The cache layer handled **144M requests** in its first week without a single consistency incident.

## Key Takeaways

- **Two-tier caching** gives you the speed of local memory with the consistency of a distributed store
- **Write-invalidate** is simpler and more predictable than write-through for most workloads
- Always have a **TTL safety net** — bugs happen, and stale data is better than poisoned data forever
- Monitor **cache hit ratio** as a first-class signal. A drop from 94% to 90% is a 66% increase in DB load
