## Problem

The product team needed real-time dashboards showing user behavior across 90-day windows. Our existing solution used batch processing with 24-hour delay and couldn't handle the growing volume of 5TB of events per day.

Queries that should take milliseconds were timing out at 30+ seconds.

## Approach

### Architecture

- **Ingestion:** Kafka as the event bus, handling 50K events/second at peak
- **Stream processing:** Apache Flink for real-time aggregations and windowing
- **Storage:** ClickHouse for columnar storage with sub-second query performance
- **API:** TypeScript service with GraphQL for flexible querying

### Key Design Decisions

- **Materialized views in Flink** for pre-computed aggregates, reducing query-time computation
- **ClickHouse's MergeTree engine** for the 90-day rolling window with automatic partitioning by day
- **Exactly-once semantics** to ensure dashboard numbers are trustworthy

## Results

| Metric | Before | After |
|--------|--------|-------|
| Query latency (P50) | 12s | 280ms |
| Query latency (P99) | 45s | 1.2s |
| Data freshness | 24 hours | < 5 seconds |
| Daily volume | 2TB | 5TB |
| Query concurrency | 5 | 200+ |

## Key Takeaways

- Columnar databases (ClickHouse) are **transformative** for analytical queries — 40x faster than Postgres for time-series aggregates
- Pre-compute aggressively: materialized views in the stream processor saved us from expensive scans
- Invest in **data quality monitoring** early — bad data in a real-time system propagates instantly
