## Problem

The API gateway was the bottleneck. Written in Node.js, it was struggling with 20K+ concurrent connections. Memory usage climbed to 4GB+ and request latency was unpredictable under load.

The rewrite needed to handle **50K concurrent connections** with **predictable sub-100ms latency** and **zero downtime migration**.

## Approach

### Why Go

- Lightweight goroutines for concurrent connection handling
- Predictable memory usage (no GC pauses under load)
- Excellent standard library for HTTP/gRPC proxying

### Architecture

The new gateway used:
- **Connection pooling** with configurable limits per upstream
- **Circuit breakers** with automatic half-open recovery
- **Request hedging** for latency-sensitive paths
- **Structured logging** with distributed trace propagation

## Results

| Metric | Node.js | Go | Improvement |
|--------|---------|----|-------------|
| Throughput | 8K req/s | 28K req/s | **3.5x** |
| Memory usage | 4.2GB | 1.1GB | **74% reduction** |
| P99 latency | 850ms | 45ms | **95% reduction** |
| Startup time | 12s | 0.3s | **97% faster** |
| Concurrent conns | 22K | 85K | **3.8x** |
